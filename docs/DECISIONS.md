# Architecture Decision Records — Agentic AI Cookbook

ADR format: Status · Context · Decision · Consequences.  
These records document the non-obvious choices. Obvious decisions (use Python, use git) are omitted.

---

## ADR-001: LLM Choice — Google Gemini 2.5 Flash

**Status:** Accepted

**Context:**  
The course needs an LLM that is free-tier accessible for learners, supports function calling,
JSON mode, and embeddings, and is reliable enough for classroom use. OpenAI GPT-4o, Anthropic
Claude, and Google Gemini were all considered.

**Decision:**  
Use **Google Gemini 2.5 Flash** as the default model, accessed via the `google-genai` SDK.
Always use 2.5 or above — never 1.5-flash or 2.0-flash.

**Consequences:**
- ✅ Free tier is generous enough for learners to complete the full course
- ✅ Supports function calling, JSON mode (`response_mime_type`), and embeddings natively
- ✅ `gemini-2.5-flash-lite` is available as a cheaper fallback for non-tool demos
- ⚠️ `gemini-2.5-flash-lite` is unreliable for function calling — must not be used for tool demos
- ⚠️ 503 errors occur under load — mitigated by the `_call()` retry wrapper in `utils/llm.py`
- ⚠️ SDK migration: `google-generativeai` is deprecated; must use `google-genai`

## ADR-002: Frameworks Come Last (Phase 10)

**Status:** Accepted

**Context:**  
Agentic AI frameworks (LangGraph, LangChain, Google ADK) provide abstractions that accelerate
development. The question was: should learners start with frameworks (faster to first demo) or
start with raw SDK calls (slower, but deeper understanding)?

**Decision:**  
Frameworks are covered only in **Phase 10**, after all 9 patterns have been implemented from
scratch against the Gemini SDK directly.

**Consequences:**
- ✅ Learners understand what each abstraction is hiding — they can debug LangGraph because they
  already know what a ReAct loop looks like in raw Python
- ✅ Learners can switch frameworks or abandon them entirely because they are not dependent
- ✅ Aligns with Anthropic's explicit recommendation: "start with the API, then add frameworks"
- ❌ Learners cannot use LangGraph for their own projects until Phase 10 — longer path to
  production tooling

## ADR-003: Streamlit as UI Layer

**Status:** Accepted

**Context:**  
The course needs an interactive UI so learners can submit queries, see tool calls, and explore
execution traces. Options were: plain Python CLI, Jupyter notebooks, Gradio, Streamlit.

**Decision:**  
Use **Streamlit** with multi-page navigation (`st.navigation()` in app.py).

**Consequences:**
- ✅ Zero JavaScript — all UI is Python, consistent with the rest of the course
- ✅ `st.session_state` provides per-user stateful memory without a backend
- ✅ Multi-page navigation groups phases naturally; sidebar auto-generates from `st.navigation()`
- ✅ `st.spinner`, `st.expander`, `st.code`, `st.progress` cover all UX needs
- ⚠️ Streamlit reruns the entire script on every interaction — state must be carefully managed
  with `st.session_state` keys and `st.rerun()`
- ⚠️ Not suitable for multi-user production deployment without session isolation

## ADR-004: Diagrams as matplotlib PNG (Not Mermaid or D3)

**Status:** Accepted

**Context:**  
Every learning page needs an architecture diagram. Options: Mermaid (text → SVG via JS),
D3.js, PlantUML, or matplotlib (Python → PNG bytes).

**Decision:**  
All diagrams are generated by **matplotlib** functions in `utils/diagrams.py`. Each function
returns `bytes` (a PNG). Pages call `st.image(diagram_fn(), ...)`.

**Consequences:**
- ✅ Pure Python — no Node.js, no browser rendering pipeline, no CDN dependency
- ✅ Diagrams are generated fresh on page load — no stale image files to maintain
- ✅ Consistent colour palette and banner style across all 20+ diagrams
- ✅ Agent Status banner (red = NOT AN AGENT, green = AGENT ✓) educates learners on what
  makes something an agent vs a workflow
- ⚠️ Emoji glyphs render as boxes on DejaVu Sans (Windows) — cosmetic, not functional
- ⚠️ Diagram code is verbose (FancyBboxPatch, annotate, etc.) — not easy to modify quickly

## ADR-005: Automatic Function Calling Must Be Disabled

**Status:** Accepted — enforced as a code standard

**Context:**  
The Gemini SDK's `client.chats.create()` has `automatic_function_calling` **enabled** by
default. When enabled, tools execute silently and `response.function_calls` returns `None`
— the tool runs but learners cannot see it happening. This destroys the pedagogical value
of the demo.

**Decision:**  
Every `chats.create()` call in the codebase explicitly sets:
```python
automatic_function_calling=types.AutomaticFunctionCallingConfig(disable=True)
```
This is enforced as a mandatory code standard, not optional.

**Consequences:**
- ✅ Every tool call is visible — learners see the function name, arguments, and result
- ✅ The execution trace expander can show the full tool call/response round-trip
- ✅ The `🔬 Execution Trace` pattern works correctly across all tool-using pages
- ❌ More boilerplate per page — the tool loop must be implemented manually
- ❌ Easy to forget in new pages — must be caught in code review

## ADR-006: Embeddings — gemini-embedding-001 at 3072 Dimensions

**Status:** Accepted

**Context:**  
Phase 5 (RAG Agent, Long-term Memory) requires text embeddings. The original implementation
used `text-embedding-004` (768-dim), which returns HTTP 404 — it is deprecated.

**Decision:**  
Use `gemini-embedding-001` exclusively. Vector dimension is **3072** (not 768).

**Consequences:**
- ✅ Embeddings work — `text-embedding-004` always 404s
- ✅ Higher-dimension embeddings produce better cosine similarity scores
- ⚠️ Any `np.zeros(768)` fallback silently produces all-zero vectors → cosine similarity
  is 0 for every document → retrieval appears to work but always returns wrong results.
  **Silent `except Exception: emb = np.zeros(N)` is banned.** Raise or log loudly.
- ⚠️ 3072-dim vectors are 4× larger in memory than 768-dim — matters at scale
